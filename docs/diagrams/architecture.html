codex-on-telegram — Architecture & Message Flow

Continue a local Codex or Claude Code session from Telegram. One owner, one Mac — a full round-trip: your message in, the model works in your project, replies and permission prompts come back in chat.

Telegram (owner only)

  • • Private chats, groups, and topics
  • • Bot API long-poll for new messages
  • • Replies + inline approval buttons sent back
  • • Locked to the owner via allow-list

Bridge daemon (shared core)

  • • channel-router maps each chat to a session
  • • conversation-engine streams the exchange
  • • delivery-layer chunks & retries replies
  • • orange = the I/O edge to Telegram & files

Permission gateway

  • • A tool call pauses on canUseTool
  • • The prompt is pushed to Telegram
  • • Approve with buttons, 1/2/3, or /perm
  • • Auto-approve optional via config

Runtime + persistence

  • • CTI_RUNTIME picks Codex, Claude, or auto
  • • The engine runs a CLI inside your project
  • • State saved as JSON under ~/.claude-to-im
  • • Sessions survive restarts — no re-bind

Supervision (macOS)

  • • launchd keeps the daemon alive
  • • Health ticks restart it on failure
  • • Secrets stay in config.env, not the plist
  • • Throttled respawn avoids crash loops
